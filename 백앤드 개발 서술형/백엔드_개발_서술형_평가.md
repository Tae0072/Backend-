# 1. Spring의 IoC/DI 원리를 설명하고, 생성자 주입이 권장되는 이유를 필드 주입·세터 주입과 비교하여 서술하시오.

## IoC (Inversion of Control) - 제어의 역전

전통적인 프로그래밍에서는 개발자가 객체를 직접 생성하고 관리합니다. IoC는 이 **제어권을 프레임워크(Spring Container)에게 위임**하는 설계 원칙입니다.

- 개발자가 `new` 키워드로 객체를 생성하는 대신, Spring이 객체(Bean)의 생명주기를 관리합니다.
- Spring Container(ApplicationContext)가 Bean을 생성·보관·소멸시킵니다.

#### DI (Dependency Injection) - 의존성 주입

IoC를 구현하는 방법 중 하나로, 객체가 필요로 하는 의존 객체를 외부(Spring Container)에서 주입해주는 방식입니다.

---

## 주입 방식 비교

**1) 필드 주입 (Field Injection)**

```java
@Service
public class UserService {

    @Autowired
    private UserRepository userRepository; // 필드에 직접 주입
}
```

- 코드가 간결하지만, `final` 선언 불가
- 스프링 컨테이너 없이 단위 테스트 불가 (순수 Java로 의존성 주입 방법 없음)
- 순환 의존성을 런타임까지 감지하지 못함

---

**2) 세터 주입 (Setter Injection)**

```java
@Service
public class UserService {

    private UserRepository userRepository;

    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

- 선택적 의존성에 적합하나, 주입 없이 객체가 생성될 수 있어 `NullPointerException` 위험
- 의존성이 변경될 수 있어 불변성 보장 불가

---

**3) 생성자 주입 (Constructor Injection) - 권장 방식**

```java
@Service
public class UserService {

    private final UserRepository userRepository; // final로 불변성 보장

    // @Autowired 생략 가능 (생성자가 하나일 경우 Spring이 자동 인식)
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

Lombok을 활용한 간결한 표현:

```java
@Service
@RequiredArgsConstructor // final 필드에 대한 생성자 자동 생성
public class UserService {

    private final UserRepository userRepository;
}
```

---

## 생성자 주입이 권장되는 이유

| 항목                   | 필드 주입  | 세터 주입  | 생성자 주입     |
| ---------------------- | ---------- | ---------- | --------------- |
| 불변성 (`final`)       | 불가       | 불가       | **가능**        |
| 순환 의존성 감지       | 런타임     | 런타임     | **컴파일 타임** |
| 단위 테스트            | 어려움     | 가능       | **쉬움**        |
| 필수 의존성 보장       | 보장 안 됨 | 보장 안 됨 | **보장**        |
| 스프링 프레임워크 의존 | 높음       | 높음       | **낮음**        |

생성자 주입은 객체 생성 시점에 모든 의존성이 반드시 주입되므로, **완전한 상태의 객체**만 생성됩니다. 또한 `final` 키워드를 사용할 수 있어 의존성이 변경되지 않음을 보장하고, 순수 Java 코드로 테스트 시 직접 Mock 객체를 생성자에 전달할 수 있어 테스트 용이성이 높습니다.

# 2. RESTful API 설계 원칙 및 CRUD 구현 (배점: 35점)

> RESTful API 설계 원칙 5가지를 설명하고, Spring Boot에서 CRUD API를 구현하는 계층별(Controller-Service-Repository) 코드를 서술하시오.

## RESTful API 설계 원칙 5가지

**1) 클라이언트-서버 구조 (Client-Server)**
클라이언트(UI)와 서버(데이터 처리)의 역할을 분리합니다. 서로 독립적으로 발전 가능하며 결합도가 낮아집니다.

**2) 무상태성 (Stateless)**
각 요청은 독립적이며, 서버는 이전 요청의 상태를 저장하지 않습니다. 필요한 모든 정보는 요청에 포함되어야 합니다. (예: JWT 토큰을 매 요청마다 헤더에 포함)

**3) 캐시 가능 (Cacheable)**
응답은 캐시 가능 여부를 명시해야 합니다. 캐시를 통해 서버 부하를 줄이고 성능을 향상시킵니다.

**4) 계층화 시스템 (Layered System)**
클라이언트는 서버에 직접 연결되어 있는지, 중간 계층(Load Balancer, Gateway)을 거치는지 알 필요가 없습니다.

**5) 균일한 인터페이스 (Uniform Interface)**
자원은 URI로 식별하고, HTTP 메서드(GET·POST·PUT·DELETE)로 행위를 표현합니다.

```
GET    /api/users       # 목록 조회
GET    /api/users/{id}  # 단건 조회
POST   /api/users       # 생성
PUT    /api/users/{id}  # 수정
DELETE /api/users/{id}  # 삭제
```

## 계층별 CRUD 구현 코드

**Entity**

```java
@Entity
@Table(name = "users")
@Getter
@NoArgsConstructor
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String username;
    private String email;

    @Builder
    public User(String username, String email) {
        this.username = username;
        this.email = email;
    }

    public void update(String username, String email) {
        this.username = username;
        this.email = email;
    }
}
```

**DTO (Request / Response)**

```java
// Request DTO
public class UserRequest {

    @NotBlank
    private String username;

    @Email
    private String email;

    public User toEntity() {
        return User.builder()
                .username(username)
                .email(email)
                .build();
    }
}

// Response DTO
public class UserResponse {

    private Long id;
    private String username;
    private String email;

    public UserResponse(User user) {
        this.id = user.getId();
        this.username = user.getUsername();
        this.email = user.getEmail();
    }
}
```

**Repository 계층**

```java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    // JpaRepository가 기본 CRUD 메서드 제공
    // save(), findById(), findAll(), deleteById() 등
}
```

**Service 계층**

```java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true) // 기본 읽기 전용 트랜잭션
public class UserService {

    private final UserRepository userRepository;

    // 전체 조회
    public List<UserResponse> findAll() {
        return userRepository.findAll().stream()
                .map(UserResponse::new)
                .collect(Collectors.toList());
    }

    // 단건 조회
    public UserResponse findById(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new RuntimeException("사용자를 찾을 수 없습니다."));
        return new UserResponse(user);
    }

    // 생성
    @Transactional
    public UserResponse save(UserRequest request) {
        User user = userRepository.save(request.toEntity());
        return new UserResponse(user);
    }

    // 수정
    @Transactional
    public UserResponse update(Long id, UserRequest request) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new RuntimeException("사용자를 찾을 수 없습니다."));
        user.update(request.getUsername(), request.getEmail());
        return new UserResponse(user); // 더티 체킹으로 자동 UPDATE
    }

    // 삭제
    @Transactional
    public void delete(Long id) {
        userRepository.deleteById(id);
    }
}
```

**Controller 계층**

```java
@RestController
@RequestMapping("/api/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    // 전체 조회
    @GetMapping
    public ResponseEntity<List<UserResponse>> findAll() {
        return ResponseEntity.ok(userService.findAll());
    }

    // 단건 조회
    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> findById(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findById(id));
    }

    // 생성
    @PostMapping
    public ResponseEntity<UserResponse> save(@RequestBody @Valid UserRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED).body(userService.save(request));
    }

    // 수정
    @PutMapping("/{id}")
    public ResponseEntity<UserResponse> update(
            @PathVariable Long id,
            @RequestBody @Valid UserRequest request) {
        return ResponseEntity.ok(userService.update(id, request));
    }

    // 삭제
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
```

# 3. JPA 영속성 컨텍스트 및 N+1 문제 (배점: 30점)

> JPA의 영속성 컨텍스트(Persistence Context)를 설명하고, N+1 문제의 원인과 해결 방법을 서술하시오.

## 영속성 컨텍스트 (Persistence Context)

영속성 컨텍스트는 **엔티티를 영구 저장하는 환경**으로, JPA가 엔티티를 관리하는 1차 캐시 공간입니다. `EntityManager`가 이를 관리하며, Spring에서는 `@Transactional` 범위 내에서 동작합니다.

**주요 특징:**

**1) 1차 캐시**

```java
@Transactional
public void example() {
    User user1 = userRepository.findById(1L); // DB 쿼리 실행
    User user2 = userRepository.findById(1L); // 1차 캐시에서 반환 (쿼리 없음)
    // user1 == user2 (동일 인스턴스)
}
```

**2) 더티 체킹 (Dirty Checking)**

```java
@Transactional
public void update(Long id, String newName) {
    User user = userRepository.findById(id).get();
    user.setUsername(newName); // 별도 save() 호출 없이 트랜잭션 종료 시 자동 UPDATE
}
```

트랜잭션 커밋 시점에 최초 조회 스냅샷과 현재 상태를 비교하여 변경된 필드만 UPDATE합니다.

**3) 쓰기 지연 (Write-Behind)**

```java
@Transactional
public void saveAll() {
    userRepository.save(new User("user1", "a@a.com")); // INSERT SQL 지연
    userRepository.save(new User("user2", "b@b.com")); // INSERT SQL 지연
    // 트랜잭션 커밋 시점에 INSERT SQL 2개를 한 번에 전송
}
```

**엔티티 생명주기:**

| 상태               | 설명                                                 |
| ------------------ | ---------------------------------------------------- |
| 비영속 (Transient) | `new`로 생성만 된 상태, 영속성 컨텍스트와 무관       |
| 영속 (Managed)     | `save()` / `findById()` 후 컨텍스트가 관리 중인 상태 |
| 준영속 (Detached)  | 트랜잭션 종료 후 컨텍스트에서 분리된 상태            |
| 삭제 (Removed)     | `delete()` 호출 후 삭제 예정 상태                    |

## N+1 문제

**원인:**

연관 관계(`@OneToMany`, `@ManyToOne`)에서 JPA가 컬렉션 조회 시 각 엔티티마다 추가 쿼리를 실행하는 문제입니다.

```java
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members;
}
```

```java
// N+1 발생 코드
List<Team> teams = teamRepository.findAll(); // 쿼리 1번 (팀 N개 조회)

for (Team team : teams) {
    team.getMembers().size(); // 팀마다 추가 쿼리 N번 실행 → 총 N+1번
}
```

팀이 10개라면 팀 조회 쿼리 1번 + 각 팀의 멤버 조회 쿼리 10번 = **총 11번** 실행됩니다.

---

**해결 방법:**

**1) Fetch Join (가장 일반적인 해결책)**

```java
// Repository
public interface TeamRepository extends JpaRepository<Team, Long> {

    @Query("SELECT t FROM Team t JOIN FETCH t.members")
    List<Team> findAllWithMembers();
}
```

```sql
-- 실행되는 SQL (1번만 실행)
SELECT t.*, m.* FROM team t INNER JOIN member m ON t.id = m.team_id
```

**2) @EntityGraph**

```java
public interface TeamRepository extends JpaRepository<Team, Long> {

    @EntityGraph(attributePaths = {"members"})
    @Query("SELECT t FROM Team t")
    List<Team> findAllWithMembers();
}
```

**3) Batch Size 설정 (IN 절로 한 번에 조회)**

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

```java
@OneToMany(mappedBy = "team")
@BatchSize(size = 100) // 개별 엔티티에 적용
private List<Member> members;
```

Batch Size 적용 시 N번의 개별 쿼리 대신 `IN (?, ?, ...)` 절로 묶어 처리합니다.

---

**해결 방법 비교:**

| 방법         | 장점                   | 단점                               |
| ------------ | ---------------------- | ---------------------------------- |
| Fetch Join   | 명확한 제어, 성능 최적 | 페이징 불가 (컬렉션 Fetch Join 시) |
| @EntityGraph | 코드 간결              | 복잡한 쿼리에 한계                 |
| Batch Size   | 페이징과 병행 가능     | 설정으로만 해결, 완전 최적화 아님  |
